### 关键关系说明

| 关系 | 类型 | 说明 |
|------|------|------|
| campus → building | 1:N | 一个校区有多栋楼 |
| building → config | 1:1 | 每栋楼有一份硬件配置 |
| building → room | 1:N | 一栋楼有多间房 |
| room → bed | 1:N | 一间房有4或6张床 |
| user → bed | 1:1 | 一个学生占一张床（通过 dorm_student_record 管理） |
| room → water_balance | 1:1 | 水电按房间计费 |
| room → electricity_balance | 1:1 | 同上 |
| user → role | N:N | 通过 sys_user_role 关联 |
| role → permission | N:N | 通过 sys_role_permission 关联 |

## 3. 基础数据模块

### 3.1 sys_campus（校区表）

校区表存储华北水利水电大学三个校区的核心信息，是整个系统数据隔离的基础维度。

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|:----:|--------|------|
| `id` | BIGINT | ✓ | AUTO | 主键ID，自增 |
| `campus_name` | VARCHAR(32) | ✓ | — | 校区名称：`龙子湖校区`、`花园校区`、`江淮校区` |
| `campus_code` | VARCHAR(16) | ✓ | — | 校区编码：`LZH`（龙子湖）、`HY`（花园）、`JH`（江淮），全系统唯一 |
| `location` | VARCHAR(64) | ✓ | '' | 所在城市：`郑州`、`信阳` |
| `sort_order` | INT | ✓ | 0 | 排序序号，越小越靠前（龙子湖=1, 花园=2, 江淮=3） |
| `status` | TINYINT(1) | ✓ | 1 | 状态：`0`-禁用, `1`-启用 |
| `create_time` | DATETIME | ✓ | NOW() | 创建时间，自动填充 |
| `update_time` | DATETIME | ✓ | NOW() | 更新时间，自动更新 |
| `is_deleted` | TINYINT(1) | ✓ | 0 | 逻辑删除：`0`-未删除, `1`-已删除 |

**索引**：
- `PRIMARY KEY (id)`
- `UNIQUE KEY uk_campus_code (campus_code)` — 校区编码唯一
- `KEY idx_status (status)` — 按状态查询

### 3.2 sys_college（学院表）

存储各学院基本信息，支持学院归属校区。

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|:----:|--------|------|
| `id` | BIGINT | ✓ | AUTO | 主键ID |
| `college_name` | VARCHAR(64) | ✓ | — | 学院名称（如：信息工程学院、水利学院） |
| `college_code` | VARCHAR(16) | ✓ | — | 学院编码，唯一 |
| `campus_id` | BIGINT | | NULL | 所属校区ID（关联`sys_campus.id`）。NULL表示跨校区学院 |
| `sort_order` | INT | ✓ | 0 | 排序序号 |
| `status` | TINYINT(1) | ✓ | 1 | 状态：`0`-禁用, `1`-启用 |
| `create_time` | DATETIME | ✓ | NOW() | 创建时间 |
| `update_time` | DATETIME | ✓ | NOW() | 更新时间 |
| `is_deleted` | TINYINT(1) | ✓ | 0 | 逻辑删除 |

## 4. 用户与权限模块

### 4.1 sys_user（用户表）

系统所有用户（学生、保洁、宿管、超级管理员）在此表统一管理，通过 `user_type` 区分角色类型。

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|:----:|--------|------|
| `id` | BIGINT | ✓ | AUTO | 主键ID |
| `username` | VARCHAR(32) | ✓ | — | 用户名。**学生=学号**，**教职工=工号**，全系统唯一 |
| `password` | VARCHAR(128) | ✓ | — | 密码，**BCrypt加密存储**，不可逆 |
| `real_name` | VARCHAR(32) | ✓ | '' | 真实姓名 |
| `phone` | VARCHAR(16) | | NULL | 手机号。绑定后可用于**登录/找回密码**，系统内唯一 |
| `email` | VARCHAR(64) | | NULL | 邮箱地址 |
| `avatar` | VARCHAR(256) | | NULL | 头像图片URL（MinIO存储） |
| `gender` | TINYINT(1) | | NULL | 性别：`0`-女, `1`-男 |
| `user_type` | VARCHAR(16) | ✓ | — | 用户类型枚举：`STUDENT`-学生, `CLEANER`-保洁, `DORM_MANAGER`-宿管, `SUPER_ADMIN`-超级管理员 |
| `campus_id` | BIGINT | | NULL | 所属校区ID（关联`sys_campus.id`）。超级管理员可为NULL；其他角色必填 |
| `college_id` | BIGINT | | NULL | 所属学院ID（关联`sys_college.id`）。学生必填 |
| `major` | VARCHAR(64) | | NULL | 专业名称（如：软件工程） |
| `grade` | VARCHAR(8) | | NULL | 入学年级（如：`2024`） |
| `last_login_time` | DATETIME | | NULL | 最后登录时间 |
| `status` | TINYINT(1) | ✓ | 1 | 账号状态：`0`-冻结（禁止登录）, `1`-正常 |
| `create_time` | DATETIME | ✓ | NOW() | 创建时间 |
| `update_time` | DATETIME | ✓ | NOW() | 更新时间 |
| `is_deleted` | TINYINT(1) | ✓ | 0 | 逻辑删除 |

**索引**：
- `UNIQUE KEY uk_username (username)` — 学号/工号唯一
- `UNIQUE KEY uk_phone (phone)` — 手机号唯一
- `KEY idx_user_type (user_type)` — 按角色类型查询
- `KEY idx_campus_id (campus_id)` — 按校区筛选（三校区隔离）
- `KEY idx_college_id (college_id)` — 按学院筛选
- `KEY idx_status (status)` — 按账号状态查询

| `perm_type` | VARCHAR(16) | ✓ | 'BUTTON' | 权限类型：`MENU`-菜单, `BUTTON`-按钮, `API`-接口 |
| `path` | VARCHAR(128) | | NULL | 前端路由路径（菜单类型时使用） |
| `component` | VARCHAR(128) | | NULL | 前端组件路径（菜单类型） |
| `icon` | VARCHAR(64) | | NULL | 菜单图标名称 |
| `sort_order` | INT | ✓ | 0 | 排序序号 |
| `status` | TINYINT(1) | ✓ | 1 | 状态 |
| `create_time` | DATETIME | ✓ | NOW() | 创建时间 |
| `update_time` | DATETIME | ✓ | NOW() | 更新时间 |
| `is_deleted` | TINYINT(1) | ✓ | 0 | 逻辑删除 |

### 4.4 sys_user_role（用户-角色关联表）

## 5. 宿舍模块

| `area` | VARCHAR(16) | ✓ | '' | 片区分组：`一区`/`二区`（龙子湖）、`南苑`/`北苑`（花园）、``（江淮填空） |

## 13. 索引设计说明

### 13.1 索引策略

| 策略 | 说明 |
|------|------|
| 主键索引 | 所有表以 `id` 为主键，使用 BTREE |
| 唯一索引 | 业务唯一字段（username, campus_code, building_code, order_no 等）加 UNIQUE 约束 |
| 联合唯一索引 | 关联表（user_role, role_permission, notice_read_record 等）防重复 |
| 普通索引 | 高频查询字段（status, campus_id, user_id, create_time）加 BTREE 索引 |
| 覆盖索引 | 水电记录表 `(room_id, record_date)` 联合索引覆盖账单查询 |

### 13.2 关键查询索引覆盖

| 查询场景 | 索引 |
|----------|------|
| 按校区查楼栋 | `idx_campus_id` ON `dorm_building` |
| 按校区查用户 | `idx_campus_id` ON `sys_user` |
| 按状态查工单 | `idx_status` ON `repair_order` |
| 保洁按日期查考勤 | `uk_cleaner_date` ON `attendance_record` |
| 房间用水账单 | `idx_room_date` ON `water_record` |
| 公告按发布时间排序 | `idx_publish_time` ON `system_notice` |
| 操作日志按时间倒序 | `idx_create_time` ON `operation_log` |
| 学生查自己的请假 | `idx_applicant_id` ON `leave_request` |

## 15. 三校区数据隔离方案

### 15.1 隔离策略

系统通过 **campus_id 字段**实现三校区数据隔离，核心规则：

| 角色 | 数据访问范围 | 实现方式 |
|------|-------------|----------|
| **学生** | 本人所在校区 | `sys_user.campus_id` 限定，所有查询自动附加 `WHERE campus_id = ?` |
| **保洁** | 被分配校区 | 同上 |
| **宿管** | 所管辖校区 | 查询时附加宿管关联的校区条件 |
| **超级管理员** | 跨校区查看全部 | 不附加 campus_id 过滤，或可通过 `data:cross_campus` 权限查看全局 |

### 15.2 实现方式

通过 MyBatis Plus 拦截器 + Sa-Token 权限注解实现：

```java
// 自动数据隔离示例（通过 MyBatis Plus 拦截器）
// 非超管用户的所有查询自动附带 campus_id = 当前用户.campus_id

// 接口层权限控制
@SaCheckPermission("data:cross_campus")  // 仅超管可跨校区
public Result getCrossCampusStats() { ... }
```

### 15.3 涉及 campus_id 的表

以下表包含 `campus_id` 字段，所有查询均需遵循三校区隔离规则：

- `sys_user` — 用户所属校区
- `sys_college` — 学院所属校区
- `dorm_building` — 楼栋所属校区
- `dorm_student_record` — 住宿校区
- `leave_request` — 请假所在校区
- `repair_order` — 报修所在校区
- `attendance_record` — 考勤校区
- `clean_task` — 任务所在校区
- `clean_work_log` — 工作所在校区
- `evaluation_config` — 评价所在校区
- `evaluation_result` — 评价所在校区
